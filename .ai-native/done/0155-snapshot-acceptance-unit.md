# Done-line 0155 — epic.environments — the snapshot acceptance unit (the spine)

Written before code, per §9.4. When this line is met, stop. The spine of `epic.environments` (bdo's confirmed arc, adm.b30f27b9d5ad), the first build of the LOCAL tier in the local→git→github phasing bdo named this session ("local, git, and github level environments for working and deployment"; the phases + work process are recorded in `epic.environments.json`). A snapshot is the unit a second party accepts — immutable, named, recorded — the way an atom's identity already IS the sha256 of its bytes. The recurring failure this closes: work strands because acceptance has hung off mutable, mid-flight PRs; a frozen named snapshot is the unit that does not strand, and the unit that joins the per-atom (content-hash) and per-PR (git) namespaces `loop/pull.py` made visible as an open gap.

This piece COMPOSES, never double-builds (§10): it sits OVER `loop/pull.py`'s `namespace_gap`, `digest`'s `atoms_on_main`/`landed_atoms` (D-13, done-line 0124), `loop/deploy.py`'s stamped-snapshot shape, and the atom content-hash identity in `reconcile.py` — it is the joining unit across them, not a second ledger. Acceptance is DERIVED from the existing pipeline's receipts (value-gate / owner-stamp), never a second write path or a second authority (D-4). The teeth are the term-economy/parity grip rule: a snapshot that LIES about its join — a frozen reference to atom bytes that have since changed, or to an atom that does not exist — is caught, not trusted. The local-deployment face (serving a snapshot on localhost, `atom.local-preview.v0`) and the git/github tiers are NOT this line.
