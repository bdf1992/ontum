# Done-line 0138 — the headless gate rail launches and leaves a trace

Written before code, per §9.4. When this line is met, stop. The branded headless gate rail (`.claude/skills/gate/gate.py launch`) could not produce a verdict when a session requisitioned it (the production-gate session, PR #286, hit this — issues #284/#285): the launched `claude -p` returned an empty result and the failure left no debuggable trace. Two root causes, found by reproduction, plus bdo's observability point ("shouldn't it have written some file?"):

1. `launch_claude` passes no `--model`, so the child `claude -p` defaults to the alias `opus`, which 404s in the headless context (`API Error: 404 model: opus`).
2. `launch_claude` runs with `cwd=ROOT`, so the repo's `UserPromptSubmit` session hooks load in the child and BLOCK the headless prompt — the process exits with `num_turns: 0`, `duration_api_ms: 0`, an empty result, and `is_error: false`. The compose() prompt is fully self-contained (it embeds the node prompt, the atom, the arc, and prior receipts), so the judge needs no repo cwd — running from a neutral cwd is safe and is the fix.
3. On any failure the rail wrote nothing, so the cause vanished (the empty tail the production-gate session saw). A run must leave a trace.

This is the gate pen's fix only; it does NOT change what a gate may decide, the §7 prompt lineage, the trust-rail issue contract, or the one-pen verdict write (D-4). Scope guard (§10): the rail is plumbing for real inference; it must not become a second place a verdict is decided.

> **Done when:** `.claude/skills/gate/gate.py`'s `launch_claude` (1) passes an explicit, configurable `--model` (env `ONTUM_GATE_MODEL`, default a valid model id) so the child no longer 404s on the `opus` alias, and (2) runs the self-contained prompt from a NEUTRAL working directory (not the repo root) so the repo's `UserPromptSubmit` hooks no longer block the headless child; AND every launch persists a run-trace FILE (the model, the argv, the prompt, raw stdout, stderr, return code) to a gitignored path under `.ai-native/log/`, written even on failure, with its path surfaced in the failure `report` and the left-open trust-rail issue comment — so a failed run is debuggable instead of vanishing; AND `tests/test_gate_rail.py` joins the suite with §10 teeth: the launch command is asserted to carry a `--model` and a cwd that is NOT the repo root, and a trace file is written on BOTH a parse-success and a parse-failure run (the failure still names the path and still raises so the issue stays open) — proven with an injected fake runner, no live spawn in the test; AND the now-fixed rail is demonstrated once end-to-end on a real atom (a live branded `value-gate.claude.v1` verdict lands through `loop.node judge` and its trust-rail issue closes), the run recorded in the session report.
