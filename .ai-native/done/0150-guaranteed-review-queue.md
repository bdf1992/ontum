# Done-line 0150 — The guaranteed review queue: landed work gets a real review, never a clog or a leak

Written before code, per §9.4. When this line is met, stop.

The root cause bdo named (2026-06-20): the landing pen (`pr.py cmd_land`) records `landed_atoms` on the merge receipt but never advances those atoms; the terminal gate (`value-confirm.claude.v1`) is an inference gate that *parks* awaiting a judge, and nothing fires that judge for landed work. So finished atoms pile up in the await-value-confirm queue forever (inflight 24/cap 8, 0 headroom — the clog). A queue is healthy only with a guaranteed consumer; the consumer is missing, so the queue is a clog. The cure (bdo, Option B): turn the clog into a real queue by giving it a guaranteed, level-triggered, idempotent processor that fires the REAL review for each queued atom — never a deterministic shortcut ("the real review must happen"). The two failure modes it must be proven against are bdo's words: it must not CLOG (an atom stuck with no consumer) and must not LEAK (an atom settled with no real review behind it).

> **Done when:** a guaranteed processor drains the value-confirm review queue, proven not to clog or leak. Concretely: (1) a `drain` verb on the gate pen (`.claude/skills/gate/gate.py`) folds the existing open review queue (`loop.summon.open_summons`) and, for each atom awaiting an admitted-real non-owner gate, fires the existing real review rail (`gate.py launch` → headless verdict → `loop.node judge`), idempotently (a re-fire is the I-2 no-op, never a double-judge) and writing nothing except through that one pen; (2) the processor is level-triggered — it re-derives the queue from the log each pass — and is wired to a guaranteed tick (an in-harness beat and/or an external scheduler) so a failed or skipped review is retried next pass, never a one-shot; the level-triggered + idempotent properties that ARE the guarantee are proven by the test; (3) `tests/test_review_queue.py` joins the suite and proves, with an injected fake review runner (no live spawn), four properties as a §10 refuse-to-fit test: a queued atom whose review returns `confirmed` reaches settled (NO CLOG); a queued atom whose review returns `missed` stays queued and does NOT settle (honest, not force-cleared); NO atom reaches settled without a `value-confirm` receipt on the log (NO LEAK); a second drain pass fires nothing new (idempotent); (4) the rail is demonstrated once end-to-end — the processor fires a real review for one genuinely-queued atom, a real verdict lands through the one pen, recorded in the session report. The processor only fires reviews; the verdict and the settle stay the gate's and the one pen's (D-4) — it never force-settles. When the queue provably drains under a guaranteed consumer and nothing settles unreviewed, stop.
