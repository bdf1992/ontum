# Done-line 0178 — The loop's guaranteed-tick heartbeat

Written before code, per §9.4. When this line is met, stop.

The clog (epic.virtual-fleet, 2026-06-21): the value-confirm review queue (done-0150) and the no-settle-on-landing rule (done-0154) shipped, and the headless review consumer was fixed (#411, rcp.merge.460). But the level-triggered loop has NO GUARANTEED DRIVER — nothing ticks it — so atoms sit at `derive:value.accepted` and never reach the real value-confirm gate; inflight piled to ~57 against a cap of 8. A level-triggered loop without a guaranteed beat is a clog by construction. `owner-stamp.bdo.v1` and `value-confirm.claude.v1` are BOTH admitted real, so a guaranteed tick is SAFE: a confirmed-arc atom advances (owner-stamp satisfied by arc confirmation), an unconfirmed atom PARKS at the real owner-stamp (never mock-stamped), and an atom reaching the terminal parks at the real value-confirm gate (no auto-settle, no leak).

> **Done when:** `loop/heartbeat.py` provides a guaranteed beat: `beat(root, drain_limit=0, by=...)` runs ONE setpoint-bounded cycle — (1) a TICK that reuses `loop.orchestrate`'s existing tick (one bounded pass; never a reimplemented pipeline), flowing confirmed-arc atoms toward value-confirm and parking unconfirmed ones with NO fake-stamp; (2) a DRAIN, only when `drain_limit>0`, that fires at most `drain_limit` real value-confirm reviews through `gate.py`'s drain (default 0 = tick-only, the cheap path). The beat is idempotent (nothing pending -> writes nothing) and level-triggered (re-derives from the log each beat), returning a summary dict (advanced, settled, drained, parked). A `--hook` mode is TICK-ONLY and fail-open (catches everything, exit 0 always) so a SessionStart beat can never raise or block a session; it is wired into `.claude/settings.json` as the in-harness guaranteed tick. The CLI `python -m loop.heartbeat [--drain-limit N] [--by WHO] [--hook]` ends with a clear done|report|needs-you. `tests/test_heartbeat.py` joins the suite and proves, with injected fakes (NO live spawn), a §10 refuse-to-fit: a confirmed-arc atom at value.accepted ADVANCES past the owner-stamp after a beat while a locally-identical UNCONFIRMED atom PARKS with NO owner-stamp receipt (the no-fake-stamp teeth); the drain fires AT MOST drain_limit reviews; a second beat with nothing pending writes/fires nothing (idempotent); and `--hook` returns exit 0 even when the underlying beat raises (fail-open). The external paced consumer (bdo's activation gesture) is documented: a Windows Task Scheduler entry running `python -m loop.heartbeat --drain-limit 3 --by heartbeat.v0` every N minutes from the repo root burns the backlog down with paced real reviews, while the in-harness tick keeps atoms flowing. When the beat guarantees the tick and the queue drains under a paced consumer, stop.
