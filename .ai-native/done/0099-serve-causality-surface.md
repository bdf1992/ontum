# Done-line 0099 — Serve Causality — the canvas and welcome mat reach the owner's phone

Written before code, per §9.4. When this line is met, stop.

The diagnosis (this session, 2026-06-16): Causality's canvas kept failing to land not because it was bad — it is good, in the established design system — but because the one judge with taste could not see it. It lived behind a `localhost` command the owner never runs, on a phone that cannot run it. Experience is judged by eye; the seeing loop was broken, so every iteration died in the dark. This line closes that loop: the experience layer is *served* where the owner already looks.

The frame (the hard rule held, not bent): the doctrine's "no network at runtime" governs the **loop** — its metabolism stays local-first (no broker, no daemon, no socket while it folds). Serving is the **experience layer**, a build-time publish of static artifacts, not a runtime seam. The loop never reaches the network; the published page is a deterministic render of committed bytes (the canvas) and a read-only fold over the committed log (the welcome mat) — cache, never truth, no second write path.

> **Done when:** A GitHub Actions workflow (`.github/workflows/pages.yml`) publishes Causality's experience layer to GitHub Pages on push to `main`: it assembles a static `_site` from the committed, self-contained canvas (`canvas.html` + `canvas.js` + `inspector.js` + `authoring.js`, served as-is) and renders the welcome mat to the site root as the landing page (`python -m causality.welcome render --out _site/index.html` — a fold over the committed log), then deploys via `configure-pages`/`upload-pages-artifact`/`deploy-pages`. The welcome mat carries an `open the canvas →` link in its header (`causality/welcome.py`), styled in the existing design tokens, so the owner's sequence welcome -> canvas is one tap. The workflow header names the experience-layer-vs-loop distinction so the hard rule is visibly preserved, not silently bent. Proven: the assemble + render steps run locally and produce a served landing page with a working canvas link (screenshot in the session report); the full suite green; a merge receipt; and, once landed, a public Pages URL the owner can open from his phone.

> **Non-example:** giving the loop a runtime network seam / daemon / socket (the hard rule is about the loop, and it holds — this is build-time only); any write path from the served page back to the log (Causality is a projection, never a second source of truth); hand-editing `welcome.html` or the assembled `_site` (generated output — the page is rendered from `welcome.py`, the canvas copied from source); routing the owner to a `localhost` command or a custom UI to "see it" (the failure this line exists to end); or treating the URL as live before it lands on `main` (it deploys on push to main, after the merge-node lands).

> **Arc:** serves the confirmed `epic.causality-surface` (adm.a675cb9d36fb) — its charter is exactly "the system gains a place to be seen and steered." Continues bdo's sequence welcome -> canvas -> tools -> paint by making welcome + canvas reachable. Rides the confirmed arc; no new owner stamp.
