# Done-line 0156 — placement folds every worktree, not just the current checkout (the concurrent-mint blind spot)

Written before code, per §9.4. When this line is met, stop. The fleet-id allocator gunked twice in one session (the 0132->0139 and 0149->0151 renumber cascades): two parallel worktrees minted the same record id because the allocator (`.claude/hooks/placement.py`, which `loop/pen.py` delegates to) folded record-id claims over all git **refs** plus only the **current checkout's** working tree — a SIBLING worktree's *uncommitted* mint was invisible. The right move when a pen gunks is to clean the pen, not scratch around it and hand it to the owner (bdo, 2026-06-20).

> **Done when:** `.claude/hooks/placement.py`'s `claims()` folds the record-id claims of **every** worktree's working tree (`git worktree list`), not just the current checkout's — so a sibling worktree's uncommitted record is counted by the allocator (`next_id`) and the collision check (`collisions`), closing the concurrent-mint race where two parallel worktrees pick one id and only the commit-check catches it. `tests/test_placement.py` proves it (and fails without the fold): a record minted-and-left-uncommitted in a sibling worktree is folded — `claims()` sees its id and `next_id()` skips past it. Fail-open is preserved: if git cannot list worktrees, it falls back to the current checkout (a broken sensor never blocks a mint).
