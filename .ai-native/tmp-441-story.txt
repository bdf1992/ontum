GitHub issue creation, closing, and commenting were ungoverned. A session typed raw `gh issue close` or `gh issue comment` and the act vanished into GitHub leaving no trace on the log. Worse, the fence rule that should have caught it (gh-issue-mutate) carried decision `prompt` — and a prompt rule is a silent no-op on the Claude surface, because only `forbidden` rules compile into command_guard's deny-list. So raw `gh issue` slipped straight through: the prompt-parity hole named in issue #412.

This PR closes that hole in two moves. First, it builds the general-issue pen at .claude/skills/issue/issue.py — the issue-side sibling of the PR pen. Its two verbs, close and comment, post through the gh CLI and then append exactly one provenance record (issue.closed / issue.commented, kind issue-governance) to the append-only log via loop.reconcile.append_line, so a governed issue mutation becomes a fact the loop can fold rather than a side effect that left quietly. The teeth: a governed close carries a reason and a signer (--by), refused at the door before any gh reach; and the record lands only after the gh call succeeds, so a close GitHub refused is never written as if it happened. Outward reach (gh) lives in the pen via subprocess, never in loop/ (network-free) — exactly like reflect.py and gate.py.

Second, it flips fence/policy.py's gh-issue-mutate rule from prompt to forbidden and re-points its justification at the new pen, then re-renders the committed .codex/ layer to match. The Claude command_guard now denies raw `gh issue close/create/comment/...`; reads (view/list/status) stay raw — they are witnessed, not gated, per the repo's read/write asymmetry. Forbidding the raw command does not block the pen: the guard watches the Bash tool, and the pen shells out to gh from inside Python where no Bash hook sees it — the same reason raw `gh pr` is forbidden while pr.py works.

tests/test_issue_pen.py proves both teeth non-vacuous: the pen records a well-formed event and refuses an empty reason / missing --by with no reach and no record; and the live command_guard, run as a subprocess, denies `gh issue close 5` while passing `gh issue view 5`, `gh issue list`, and `gh issue status` — a guard that denied everything would fail the read assertions, and the old prompt no-op would fail the close assertion. The full suite is green, and tests.test_fence stays green (the committed .codex bytes equal a fresh render).
